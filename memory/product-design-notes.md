## 核心理念

> **产品设计不是画图，是理解用户，解决问题。好的设计，用户感觉不到设计，只觉得好用。**

---

## 设计流程：双钻模型 (Double Diamond)

```
发现问题(发散) → 定义问题(收敛) → 开发方案(发散) → 交付验证(收敛)
```

1. **发现** - 理解问题，用户研究
2. **定义** - 聚焦问题，明确需求
3. **开发** - 探索方案，原型设计
4. **交付** - 验证方案，落地实施

---

## 核心知识体系

### 用户研究方法
- 用户访谈 → 需求/痛点
- 问卷调查 → 画像/偏好
- 焦点小组 → 反馈/洞察
- 可用性测试 → 体验问题/优化点
- 数据分析 → 使用模式/转化漏斗

### 需求分析方法
| 方法 | 应用场景 |
|------|----------|
| KANO 模型 | 需求优先级（基本/期望/兴奋） |
| MoSCoW 法则 | 版本规划（Must/Should/Could/Won't） |
| RICE 评分 | 需求排序（Reach/Impact/Confidence/Effort） |
| 用户故事地图 | 功能规划 |
| JTBD | 需求洞察 |

### 交互设计原则
1. **可见性** — 功能应该可见
2. **反馈** — 操作后有反馈
3. **约束** — 限制错误操作
4. **一致性** — 保持界面一致
5. **容错** — 允许犯错和恢复

### 信息架构原则
清晰性 → 一致性 → 可寻性 → 可扩展性 → 简洁性

## 实战能力

### 1. PRD 文档结构
背景 → 功能需求 → 用户故事 → 功能详情 → 数据需求 → 成功标准

### 2. 原型设计流程
草图(Sketch) → 线框图(Wireframe) → 可交互原型(Prototype) → 设计稿(Mockup)

### 3. 用户体验地图
阶段（认知→考虑→使用→复购→推荐）× 维度（行为/想法/情绪/痛点/机会）

### 4. A/B 测试
假设 → 变体设计 → 指标确定 → 样本量计算 → 运行测试 → 结果分析

### 5. 设计评审清单
目标对齐 → 用户体验 → 技术可行 → 商业价值 → 下一步

---

## 常见陷阱 ⚠️
- 自嗨设计（自己觉得好）
- 过度设计（功能太多）
- 忽视边界情况
- 不考虑技术限制
- 不验证就上线
